ファイルサーバー移行ガイド

既存のファイルサーバーから DirectCloud への移行を、段階的に進める

DirectCloudを契約したものの、既存のファイルサーバーやNASにデータが残ったままになっていませんか？
二重管理の状態が続くと、どちらに最新ファイルがあるかわからなくなり、業務効率が下がります。このガイドでは、ファイルサーバーからDirectCloudへの移行を段階的に進める方法を学びます。

移行は大きく5つのSTEPで進めます。
現状の整理から移行対象の選定、フォルダ設計、データ移行、そして切り替えまで、順番に見ていきましょう。

こんな方向け:
・ ファイルサーバーやNASが社内に残っていて、DirectCloudと二重管理になっている
・ 移行したいが、どこから手をつけるか・どのくらい時間がかかるか見えない

STEP 1 — 現状を整理する

まず既存ファイルサーバーの全体像を把握します

移行の第一歩は、既存ファイルサーバーの現状を把握することです。
「全部移行する」と決める前に、まず何がどこにあるかを可視化しましょう。

確認すべき項目

棚卸しチェックリスト

  • 総容量：ファイルサーバー全体で何GB（何TB）使っているか
  • フォルダ数：第1階層・第2階層のフォルダ構成を一覧化
  • 利用頻度：毎日使う / 月1回程度 / アーカイブ（ほぼ触らない）
  • アクセス権の構造：誰がどのフォルダにアクセスできるか

なぜ可視化が重要か

1

優先順位がつけられる

利用頻度の高いフォルダから移行すれば、業務への効果が早く出る

2

不要データを除外できる

放置されたフォルダや重複ファイルを移行対象から外せる

3

スケジュールが立てられる

容量やフォルダ数がわかれば、移行にかかる時間の見積もりが可能に

移行の目安

  • 利用頻度が高いフォルダから優先的に移行する
  • 1年以上アクセスのないフォルダは移行ではなく、アーカイブとして別途検討する
  • フォルダ単位で「最終更新日」を確認すると、実際に使われているフォルダと放置されたフォルダが見分けられる
Tip: フォルダ単位で「最終更新日」を確認すると、実際に使われているフォルダと放置されたフォルダが見分けられます。Windowsのエクスプローラーで「更新日時」列を表示して並べ替えるだけでも、概要をつかめます。

STEP 2 — 移行対象を選定する

全部一気に移行しない。「フェーズ分け」が成功の鍵です

現状が把握できたら、次は移行対象を選定します。
一度にすべてを移行しようとするとトラブルの原因になります。フェーズ分けで段階的に進めましょう。

推奨フェーズ分け

1

フェーズ1: 毎日使うフォルダ

部署共有フォルダ、プロジェクト進行中のフォルダなど、日常的にアクセスするもの

2

フェーズ2: 定期的に使うフォルダ

月次レポート、テンプレート、定型書類など、定期的に参照するもの

3

フェーズ3: アーカイブ

過去プロジェクト、法定保存文書など、頻繁にはアクセスしないもの

移行しないもの

除外対象

  • 個人PC内のローカルファイル：ファイルサーバー移行とは別管理
  • 重複ファイル：同じファイルが複数フォルダにあれば整理してから移行
  • 一時ファイル：tmpファイル、作業途中の下書きなど

移行対象リストの作成

  • Excelで管理すると全体像が把握しやすい
  • 列の例: フォルダ名 / 容量 / 優先度 / 担当者 / 移行予定日
  • フェーズごとに色分けすると、進捗管理がしやすくなる
Tip: 移行対象リストをExcelで作成しておくと、担当者の割り当てや進捗管理がしやすくなります。フォルダ名・容量・優先度・担当者・移行予定日の列を作り、フェーズごとに整理しましょう。

STEP 3 — DirectCloud側のフォルダを設計する

既存の構成をそのまま移すのではなく、整理するチャンスです

移行先のフォルダ構成は、既存サーバーをそのままコピーする必要はありません。
むしろ、移行はフォルダ構成を見直す絶好のタイミングです。

フォルダ設計のポイント

📁

第1階層の設計

部署単位またはプロジェクト単位で構成。深すぎる階層は3〜4階層までに整理

👥

アクセス権の設計

グループ単位でアクセスレベルを設定。個人単位の設定は最小限に

📝

命名規則の策定

フォルダ名・ファイル名の統一ルール（日付形式、バージョン管理など）を決める

推奨フォルダ構成の例

  1. 第1階層：部署名（営業部、総務部、開発部など）
  2. 第2階層：業務カテゴリ（案件管理、月次報告、共有テンプレートなど）
  3. 第3階層：年度やプロジェクト名（2026年度、ProjectAなど）
  4. 第4階層以降：必要に応じて。ただし深くしすぎない

公式マニュアル

  • 管理者マニュアル：https://help.directcloud.jp/s/admin-manual

フォルダ構成の見直しは今がチャンス

  • フォルダ構成の見直しは、移行時にしかできないタイミング
  • 後から変更するのは、ファイルの移動やリンクの修正が必要になり大変
  • このステップに時間をかける価値がある
Tip: アクセス権はグループ単位で設定するのが管理効率の面で推奨です。個人単位で細かく設定すると、異動や退職のたびに修正が必要になります。

STEP 4 — データを移行する

プランに応じた移行方法を選び、計画的にデータを移行します

フォルダ設計ができたら、いよいよデータの移行です。
契約プランによって利用できる移行方法が異なります。自社の状況に合った方法を選びましょう。

移行方法の比較

方法 向いている場面 注意点
Webアプリからアップロード 少量のフォルダ（数GB程度） 1ファイル最大10GB（スタンダードプラン）
DirectCloudドライブ経由 大量ファイルの移行 エクスプローラー感覚で操作可能。ネットワーク速度に依存
DCMigrator 大規模移行 自己利用はエンタープライズプランのみ。パートナー経由の初期移行は全プラン対応（別途費用）

移行中のルール

並行運用時の基本ルール

  • ファイルサーバーは読み取り専用にする（新規保存・編集を禁止）
  • 新規ファイルはDirectCloudに保存するルールを全社に周知
  • 移行対象フォルダごとに完了チェックを行い、漏れを防ぐ

DCMigratorについて

  • ユーザー自身での利用はエンタープライズプランのみ
  • パートナー（販売代理店）による初期移行作業であれば全プランで利用可能（別途費用が発生します）
  • 移行元のファイルサーバーにエージェントをインストールして実行する
  • 大規模なデータ（数百GB〜TB規模）の移行に適している
Tip: 一度に大量のファイルをアップロードするとネットワーク帯域に影響する場合があります。業務時間外や週末に実施するのが安全です。DirectCloudドライブを使えばエクスプローラーのドラッグ&ドロップ感覚で移行できます。

STEP 5 — 切り替えと旧サーバーの扱い

移行完了後の並行運用期間と、旧サーバー停止の進め方を確認します

データ移行が完了しても、すぐにファイルサーバーを停止しないのがポイントです。
並行運用期間を設け、段階的にアクセスを閉じていきましょう。

並行運用の進め方

1

並行運用期間を設ける

推奨: 2〜4週間。DirectCloud側のデータが正しいか、業務フローが回っているかを確認

2

段階的にアクセスを閉じる

まず書き込み禁止 → その後読み取りも閉じる。いきなり全停止しない

3

旧データのバックアップ保持

旧サーバーのデータは即削除せず、3〜6ヶ月はバックアップとして保持

並行運用中に確認すること

  1. DirectCloud側のデータが正しく移行されているか（ファイル数・フォルダ構成）
  2. アクセス権が想定通りに設定されているか（各部署のメンバーで確認）
  3. 業務フローが問題なく回っているか（日常業務で支障がないか）
  4. ユーザーからの問い合わせ内容を記録し、共通の課題がないか確認

切り替え後に多い問い合わせ

  • 「ファイルが見つからない」という声が増えやすい
  • DirectCloudの検索機能の使い方を事前に案内しておくと問い合わせが減る
  • 旧サーバーのフォルダパスとDirectCloudのフォルダ構成の対応表を共有すると効果的
Tip: 切り替え後に「ファイルが見つからない」という問い合わせが増えがちです。DirectCloudの検索機能やフォルダ構成の対応表を事前に案内しておくと、ユーザーの混乱を防げます。

Q5. データ移行完了後、旧ファイルサーバーの扱いとして最も適切なのは？

A. 移行完了と同時に即停止・データ削除する

B. 並行運用期間（2〜4週間）を設け、段階的にアクセスを閉じる

万が一のためにファイルサーバーをずっと稼働させておく
